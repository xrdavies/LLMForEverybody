## Квантовая механика: Бог играет в кости

***Кот Шредингера***

В закрытую коробку помещают кота, стеклянную колбу с газом цианистого водорода и вещество радий-225. Когда монитор внутри коробки обнаруживает распавшуюся частицу, он разбивает колбу и убивает кота. Согласно копенгагенской интерпретации квантовой механики, спустя некоторое время эксперимента кот находится в суперпозиции: одновременно жив и мертв. Но если экспериментатор заглянет внутрь коробки, он увидит либо живого кота, либо мертвого, а не кота в двух состояниях сразу.

![alt text](../../../08-第八章-大模型企业落地/assest/大模型落地难点之输出的不确定性/1.png)

Эйнштейн считал, что это неправильно, и говорил: "Я не верю, что Бог играет в кости".

Главный вопрос спора в этом мысленном эксперименте: судьба кота уже определена до открытия коробки или определяется только в момент открытия? Школа классической физики считает, что до открытия коробки кот уже либо жив, либо мертв; это базовая интуиция. Копенгагенская школа считает, что пока коробка не открыта, кот находится и в живом, и в мертвом состоянии, а его судьба определяется только при открытии; это свойство микрочастиц.

Копенгагенская школа считает, что фундаментальная ошибка эксперимента с котом в том, что люди применяют здравый смысл макромира к свойствам микромира.

![alt text](../../../08-第八章-大模型企业落地/assest/大模型落地难点之输出的不确定性/2.png)

***Вывод академического сообщества***

- **Микромир**: Бог играет в кости; в квантовом мире, похоже, действительно существует случайность и высокая вероятность вероятностного поведения.
- **Макромир**: Бог не играет в кости; всем управляют закономерности, наука есть наука.


В разработке software, особенно после входа больших моделей в workflow, тоже есть различие между deterministic и probabilistic.


## Традиционная разработка software: Deterministic

> Примечание: здесь мы не рассматриваем software с намеренно случайным output, например random number generator.

В data flow разработки software мы обычно встречаем три типа проблем: белая дыра, черная дыра и серая дыра. Если решить эти три типа проблем, data flow заработает. Но скрытая предпосылка в том, что определенный input дает определенный output, а software должен гарантировать **идемпотентность**: независимо от того, в который раз подается один и тот же input, output должен быть одинаковым. В реальной разработке отсутствие **идемпотентности** означает, что фактически input в разные разы был разным, просто вы думали, что он одинаковый.

Для конкретного input output software определен. Это и есть deterministic.

  - Белая дыра: input отсутствует, но output есть
  - Черная дыра: input есть, но output отсутствует
  - Серая дыра: input недостаточен для получения output


## Разработка приложений на больших моделях: Probabilistic

Генерация token большой моделью по своей сути основана на вероятности. Даже при полностью одинаковом input output может отличаться.

**Greedy**

Большая модель **не использует** эту стратегию!

Выбор слова/token с наибольшей вероятностью называется greedy algorithm. Но greedy algorithm приводит к повторяемости output, потому что выбирает только самое вероятное слово и не учитывает вероятность других слов.

![alt text](../../../08-第八章-大模型企业落地/assest/大模型落地难点之输出的不确定性/4.png)

**Sampling**

Большая модель **использует** эту стратегию!

Случайная стратегия выбирает один token из распределения вероятностей всех tokens. Здесь вероятность выбора "cake" равна 20%, но фактически выбран "banana".

![alt text](../../../08-第八章-大模型企业落地/assest/大模型落地难点之输出的不确定性/5.png)

***Top-k***

После применения случайной стратегии выбирается один token из первых k результатов.

![alt text](../../../08-第八章-大模型企业落地/assest/大模型落地难点之输出的不确定性/6.png)

***Top-p***

Случайная стратегия выбирает output из непрерывного набора кандидатов, отсортированных по вероятности сверху вниз, при этом накопленная вероятность <= p.

![alt text](../../../08-第八章-大模型企业落地/assest/大模型落地难点之输出的不确定性/7.png)

## Bitter Lessons

Неопределенность - обоюдоострый инструмент.

С одной стороны, генерация становится разнообразнее и в некоторой степени повышает качество output: текст получается богаче, разнообразнее и интереснее.

С другой стороны, когда ожидается строгий output вроде JSON, SQL или кода, такая неопределенность иногда неприемлема.

Неопределенность больших моделей определяется свойствами модели, а не ошибками реализации. При разработке LLM-приложений все участники полного процесса должны заранее учитывать эту неопределенность, чтобы лучше обрабатывать edge cases.


> FAQ: а к какому типу относится прежнее machine learning/deep learning?

> В общем случае их output использует Greedy strategy: выбирается вариант с максимальной вероятностью. Для определенного input будет определенный output. Если два output отличаются, значит input на уровне модели тоже был разным.
