## 1. Что такое галлюцинации?

Галлюцинации большой модели, если говорить просто, - это "уверенная чепуха".
В формулировке статьи `A Survey on Hallucination in Large Language Models`[<sup>1</sup>](#refer-anchor-1), это явление, при котором сгенерированный моделью content не соответствует фактам реального мира или input пользователя.
Исследователи делят галлюцинации больших моделей на фактические галлюцинации (Factuality Hallucination) и галлюцинации верности контексту (Faithfulness Hallucination).

![alt text](../../../08-第八章-大模型企业落地/assest/大模型落地难点之幻觉/2.png)

### Фактические галлюцинации

Это ситуация, когда сгенерированный моделью content не соответствует проверяемым фактам реального мира.

Например, если спросить модель: "Кто первым ступил на Луну?", она может ответить: "Charles Lindbergh первым высадился на Луне в миссии Moon Pioneer 1951 года". На самом деле первым человеком на Луне был Neil Armstrong.

Фактические галлюцинации можно дополнительно разделить на factual inconsistency (противоречие информации реального мира) и factual fabrication (полностью выдуманные сведения, которые нельзя проверить по реальной информации).

### Галлюцинации верности контексту

Это ситуация, когда сгенерированный моделью content не соответствует пользовательской инструкции или контексту.

Например, модель просят резюмировать новости октября этого года, а она начинает говорить о событиях октября 2006 года.

Галлюцинации верности контексту тоже можно разделить на три типа: instruction inconsistency (output отклоняется от пользовательской инструкции), context inconsistency (output не соответствует контекстной информации) и logical inconsistency (несоответствие между шагами рассуждения и финальным ответом).


## 2. Как работать с галлюцинациями

В production мы не любим hallucinations: нам нужны точные и корректные ответы.

В реальном внедрении production-систем мы постепенно применяем следующие стратегии, чтобы повысить точность и снизить галлюцинации:

| Стратегия | Сложность | Требования к данным | Рост точности |
| :--- |:----:| :----: |---: |
| Prompt engineering | низкая | нет | 26% |
| Self-reflection | низкая | нет | 26-40% |
| Few-shot learning (with RAG) | средняя | мало | 50% |
| Instruction Fine-tuning | высокая | средние | 40-60% |


## 3. Prompt Engineering

Prompt Engineering - это искусство и наука оптимизации prompts для получения эффективного output. Оно включает проектирование, написание и изменение prompts, чтобы направлять AI-модель к генерации качественных, релевантных и полезных ответов.

## 4. Self-reflection

***Пример K12***

Укажите модели найти собственное решение, прежде чем спешить с выводом: семь раз подумай, один раз ответь.

Иногда, когда мы явно просим модель рассуждать от first principles перед выводом, получаются лучшие результаты.

Предположим, мы хотим, чтобы модель оценила решение математической задачи учеником.

Самый очевидный способ решить это - просто спросить модель, правильно ли решение ученика. Но действительно ли так получится правильный ответ?

**Неверный пример**

![alt text](../../../08-第八章-大模型企业落地/assest/大模型落地难点之幻觉/4.png)

**Правильный пример**

![alt text](../../../08-第八章-大模型企业落地/assest/大模型落地难点之幻觉/5.PNG)

![alt text](../../../08-第八章-大模型企业落地/assest/大模型落地难点之幻觉/6.PNG)

Self-reflection часто используется в больших моделях для уменьшения галлюцинаций, то есть информации, которая звучит правдоподобно, но фактически неточна или бессмысленна. Через интерактивный self-reflection можно использовать multitask-способности LLMs: генерировать, оценивать и постоянно улучшать знания, пока не будет достигнут удовлетворительный уровень фактичности.

Как встроить self-reflection в workflow? Рассмотрим пример NL2SQL[<sup>2</sup>](#refer-anchor-2):

### Первая интеракция
```python
question = ''
prompt = f'{question}'
plain_query = llm.invoke(prompt)
try:
    df = pd.read_sql(plain_query)
    print(df)
except Exception as e:
    print(e)
```
### reflection

```python
reflection = f"Question: {question}. Query: {plain_query}. Error:{e}, so it cannot answer the question. Write a corrected sqlite query."
```

### Вторая интеракция

```python
reflection_prompt = f'{reflection}'
reflection_query = llm.invoke(reflection_prompt)
try:
    df = pd.read_sql(reflection_query )
    print(df)
except Exception as e:
    print(e)
```

Через reflection мы можем постоянно улучшать наш вопрос, пока не получим нужный ответ.


## 5. Few-shot learning (with RAG)

### Few-shot learning

Добавить в prompt небольшое число примеров, чтобы помочь большой модели лучше понять задачу.

![alt text](../../../08-第八章-大模型企业落地/assest/大模型落地难点之幻觉/7.png)

### RAG

Retrieval-Augmented Generation (RAG) объединяет большую языковую модель (LLM) и систему information retrieval, чтобы повысить точность и релевантность генерируемого текста. Метод позволяет модели до генерации ответа извлечь релевантную информацию из авторитетной базы знаний, обеспечивая актуальность и профессиональность output без переобучения самой модели.

RAG важен потому, что решает несколько ключевых проблем LLM: предоставление ложной информации, генерацию устаревших сведений, зависимость от неавторитетных источников и неточные ответы из-за смешения терминов. Введение RAG позволяет извлекать информацию из авторитетных и заранее определенных источников знаний, усиливает контроль над сгенерированным текстом и повышает доверие пользователей к AI-решениям.

![alt text](../../../08-第八章-大模型企业落地/assest/大模型落地难点之幻觉/8.PNG)

### Few-shot with RAG

В методе на базе RAG мы можем динамически выбирать наиболее релевантные примеры для few-shot learning по сходству между query и кандидатами-примерами. Такой подход не только повышает точность генерации, но и делает модель более гибкой и умной при обработке разных типов query.

Конкретнее, RAG оценивает сходство между query и кандидатами-примерами и извлекает из базы кандидатов самые релевантные cases. Эти выбранные cases используются как примеры для few-shot learning, помогая модели лучше понимать и генерировать content, связанный с query. Благодаря динамическому выбору модель может гибко подстраивать используемые примеры под конкретные потребности каждого query, обеспечивая более эффективное обучение и генерацию.

Преимущество метода в том, что он максимально использует существующую knowledge base, динамически реагирует на разные query и сильно повышает практичность и точность модели.


## 6. Instruction Fine-tuning

В production fine-tuning - самый сложный путь, потому что[<sup>2</sup>](#refer-anchor-2):
- нужно больше вычислений для той же точности, иногда в 10 000 или 1 000 000 раз больше;
- невозможно эффективно распараллелить на нескольких GPU, из-за чего теряется много свободных GPU-ресурсов;
- на реальных use cases легко ломается, невозможно непрерывно fine-tune и выполнять inference в production;
- LLM не улучшается, ее трудно адаптировать под каждый use case, schema и dataset;
- трудно использовать и масштабировать из-за GPU и memory issues;
- совмещать fine-tuning и inference легко с ошибками.

При этом нужно учитывать: как получить **качественные данные**?

Ниже простой cheat-sheet по шагам получения данных[<sup>2</sup>](#refer-anchor-2):
- у вас больше данных, чем вы думаете;
- сначала проведите инвентаризацию данных, которые у вас есть;
- обычно у вас много данных, просто их формат не соответствует требованиям LLMs;
- вы не хотите вручную размечать данные или чистить их для переформатирования;
- но это не страшно: LLM может помочь, если вы зададите правильный формат.

## Итоги

С помощью четырех стратегий выше можно эффективно повысить точность большой модели и снизить появление галлюцинаций. В production мы можем выбирать подходящую стратегию по ситуации или комбинировать несколько стратегий для лучшего результата.
