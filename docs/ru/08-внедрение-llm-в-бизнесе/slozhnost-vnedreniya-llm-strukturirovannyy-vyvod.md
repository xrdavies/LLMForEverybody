## Приложения прежде всего

На World Artificial Intelligence Conference (WAIC) 2023 ключевой темой была "битва ста моделей"; в этом году ключевое слово WAIC - "приложения прежде всего". Если посмотреть на горячие темы форумов этого года, будь то embodied intelligence или AI Agent, все они указывают на вертикальные приложения AI-технологий во главе с большими моделями в разных сценариях.

Если смотреть на **output** модели, у приложений больших моделей есть две парадигмы:

***Output в виде неструктурированных данных***: QA-боты, интеллектуальная поддержка клиентов или upstream input для другой большой модели относятся к этой парадигме. Техническая архитектура - domain LLM + RAG, требований к формату output нет.

***Output в виде структурированных данных***: когда нужно встроить большую модель в workflow, особенно уже существующий, модель должна взаимодействовать с исходными компонентами workflow. В такой ситуации мы ожидаем от большой модели структурированный output, например JSON.

## Как выводить JSON

Нужно подсказать большой модели в prompt. Конкретный prompt похож на:

 The output should be formatted as a JSON instance that conforms to the JSON schema below.

## JSON output в OpenAI

На прошлогоднем DevDay OpenAI представила JSON Schema - инструмент, созданный для разработчиков, чтобы помогать им строить более надежные приложения. Хотя JSON Schema повышает точность генерации валидного JSON output, она не гарантирует, что response полностью соответствует конкретной schema. Чтобы преодолеть это ограничение, OpenAI дальше выпустила в API функцию structured outputs, которая гарантирует точное соответствие output модели JSON Schema, предоставленной разработчиком.

Преобразование неструктурированного input в структурированные данные - одно из ключевых применений больших моделей (LLMs). Разработчики используют OpenAI API для создания мощных интеллектуальных ассистентов: они могут через function calling получать данные, отвечать на вопросы, извлекать структурированные данные для data entry и строить многошаговые agent workflows, позволяя LLMs выполнять реальные задачи.

Раньше разработчики решали ограничения LLMs в генерации структурированных данных через open-source инструменты, тщательно написанные prompts и множество проб разных requests, чтобы output модели мог без проблем состыковаться с их системами.

Теперь structured outputs решает эти проблемы: оно принуждает модель следовать указанной разработчиком schema и дополнительно обучает модель лучше понимать сложные schemas.

## Corner/Edge Case

### Почему важно учитывать edge cases

Учет edge cases - ключ к надежности системы. В задачах вроде LeetCode полнота отделяет хорошее решение от обычного. Точно так же при интеграции большой модели в workflow нужно заранее предусмотреть и обработать все возможные исключения.

### Вероятностная природа больших моделей

Даже если большая модель достигает точности 99,99%, теория вероятностей говорит: с ростом числа запусков даже очень маленькая вероятность отказа рано или поздно приведет к сбою. За десять тысяч запусков хотя бы один сбой почти неизбежен.

### Ожидания к output большой модели

В нашем сценарии мы полагаемся на то, что большая модель выдаст output, соответствующий заранее заданной JSON Schema. Любой неожиданный результат может прервать workflow и повлиять на стабильность и эффективность всей системы.

### Типы ошибок

Типы ошибок могут быть разными. Главным образом стоит обращать внимание на такие случаи:

- валидность JSON: невалидная JSON-структура может привести к ошибке parsing и повлиять на дальнейшую обработку данных;

- уровни вложенности JSON: большая модель иногда может без явной инструкции добавить или убрать уровень вложенности, разрушив ожидаемую структуру данных;

- проблемы JSON key: результат key не соответствует договоренности, или при отсутствии value соответствующий key тоже может неожиданно исчезнуть;

- ошибка соответствия JSON key-value: когда keys похожи, может возникнуть mismatch key-value, то есть value ошибочно связывается не с тем key.

### Решение

Все эти ситуации могут привести workflow к сбою. Поэтому хороший способ - выполнить reflection перед выдачей результата, а более надежный способ - добавить проверки.

- reflection: перед output модели можно через reflection проверить структуру output и убедиться, что она соответствует ожидаемой JSON Schema; этот способ подходит для сценариев без жестких требований к latency;

- hard code validation: после output модели можно через validation проверить структуру output и убедиться, что она соответствует ожидаемой JSON Schema; этот способ подходит для сценариев с более жесткими требованиями к latency.
