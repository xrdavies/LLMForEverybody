Полезный материал: практическое руководство по внедрению enterprise RAG

В enterprise-данных многое поступает из разных типов документов, например PDF, Word-документов, электронных писем и веб-страниц. Нам нужно сосредоточиться на двух этапах: Load & Process и Split/Chunking.

## 1. Что такое RAG?
Retrieval-Augmented Generation (сокращенно RAG, генерация с дополнением через поиск) повышает точность и релевантность генерируемого текста за счет объединения больших языковых моделей (LLM) и систем информационного поиска. Этот подход позволяет модели перед генерацией ответа сначала извлечь релевантную информацию из авторитетной базы знаний, тем самым обеспечивая актуальность и профессиональность результата без повторного обучения самой модели.

Технология RAG важна потому, что решает ряд ключевых проблем LLM: выдачу ложной информации, генерацию устаревших сведений, опору на неавторитетные источники, а также неточные ответы из-за путаницы в терминологии. Благодаря RAG информацию можно извлекать из авторитетных и заранее определенных источников знаний, что усиливает контроль над генерируемым текстом и одновременно повышает доверие пользователей к AI-решениям.

## 2. Сложности внедрения enterprise RAG

В enterprise-данных многое поступает из разных типов документов, например PDF, Word-документов, электронных писем и веб-страниц. Нам нужно сосредоточиться на двух этапах:

- Load & Process, то есть A на схеме выше, — это процесс загрузки данных. В реальных приложениях формат и структура данных сильно различаются. Поэтому эффективная загрузка и обработка таких данных — очень сложная задача.

- Split/Chunking, то есть B на схеме выше, — это процесс разделения данных на несколько частей. В реальных приложениях данные обычно неструктурированы, поэтому их нужно аккуратно разделять и обрабатывать, чтобы модель могла лучше их понимать и использовать.

## 3. Информация, которую нужно load
Помимо получения текстовой информации из документа, важной структурной информацией являются и другие данные, например имя файла, номер страницы и т. д. В практике RAG нам нужно извлекать всю эту информацию, чтобы лучше понимать и обрабатывать данные.

**Элементы документа**: базовые составляющие документа, которые можно использовать для разных RAG-задач, например фильтрации и разбиения на блоки:
- Заголовок
- Повествовательный текст
- Элемент списка
- Таблица
- Изображение

**Метаданные элементов**: дополнительная информация об элементах, которую можно использовать для фильтрации в hybrid search и для определения источников ответа:
- Имя файла
- Тип файла
- Номер страницы
- Раздел

> Примечание: если вы не понимаете, что такое hybrid search, ничего страшного, дальше мы подробно это разберем.


## 4. Обработка данных
Обработка данных необходима, но сложна, главным образом потому что:
- Подсказки по содержимому: разные типы документов дают разные подсказки о типах элементов (визуальные, markdown);
- Необходимость стандартизации: чтобы обрабатывать содержимое из разных типов документов, его нужно стандартизировать;
- Разные способы извлечения: разные форматы документов могут требовать разных методов извлечения;
- Понимание метаданных: во многих случаях для извлечения метаданных нужно понимать структуру документа.

Нам нужно преобразовать разные типы документов (PDF, Word, EPUB, MarkDown и т. д.) в единый формат, чтобы модель могла лучше их понимать и обрабатывать. Один простой и эффективный способ — преобразовать их в формат JSON.

У формата JSON есть следующие особенности:

- Структура распространена и проста для понимания
- Это стандартный HTTP-ответ
- Может использоваться во многих языках программирования
- Может быть преобразован в JSONL для streaming-сценариев

Ниже приведен пример преобразования в JSON:
```Json
[
    {
        "element_id":"bff1fd0ec25e78f1224ad7309a1e79c4",
        "metadata":{
        "filename": "CoT.pdf",
        "filetype":"application/pdf",
        "languages":[
            "eng"
        ],
        "page_number":1,
        },
        "text":"B All Experimental Results",
        "type": "Title"
    },
    {
        "element_id":"ebf8dfb149bcbbd8c4b7f9a7046900a9",
        "metadata":{
        "filename": "CoT.pdf",
        "filetype":"application/pdf",
        "languages":[
            "eng"
        ],
        "page_number":1,
        },
        "text": "This section contains tables for experimental results for varying models and model sizes, on all benchmarks, for standard prompting vs. chain-of-thought prompting.",
        "type": "NarrativeText"
    }
]
```

Для разработчика это означает необходимость найти фреймворк, который способен обрабатывать разные типы документов и преобразовывать их в формат JSON.
Некоторые типы документов (например, HTML, Word Docs и Markdown) содержат информацию о форматировании и могут предварительно обрабатываться парсерами на основе правил; но для PDF или документов-изображений нужны другие технологии обработки. Такие технологии обычно не являются open source: их приходится покупать или разрабатывать самостоятельно.


## 6. Семантический поиск и hybrid search
Для большинства людей семантический поиск, Semantic Search, хорошо знаком: его цель — по заданному входному тексту найти в корпусе документов семантически похожее содержимое, чтобы добавить его в Prompt.
Но семантический поиск не всемогущ. У него есть ограничения, например:
- Слишком много результатов поиска: при большом количестве документов семантически похожих совпадений оказывается слишком много;
- Самая свежая информация: пользователю может быть нужна самая свежая информация, а не просто семантически наиболее похожая;
- Потеря важной информации: теряется важная структурная информация документа, связанная с поиском, например заголовки, номера страниц и т. д.

Гибридная стратегия: hybrid search — это поисковая стратегия, которая сочетает семантический поиск с другими технологиями информационного поиска, например фильтрацией и keyword search. Параметры фильтрации берутся из метаданных документа.


## 7. Chunking

Векторной базе данных нужно разделить документ на блоки, чтобы выполнять поиск и генерировать подсказки. В зависимости от способа chunking одного и того же документа один и тот же запрос будет возвращать разное содержимое.

**Блоки одинакового размера**: самый простой метод — разделить документ на блоки примерно одинакового размера. Это приводит к тому, что похожее содержимое оказывается разделено между несколькими блоками.

**Разбиение по атомарным элементам**: распознав атомарные элементы, можно формировать блоки путем объединения элементов, а не разрезания исходного текста. Так получаются более связные блоки.

***Шаги***:

1. Разделение: сначала разбить документ на атомарные элементы;
2. Объединение элементов в блоки: добавлять элементы документа в блок, пока не будет достигнут порог по символам или токенам.
3. Применение условий разрыва: применять условия начала нового блока, например когда мы достигаем нового элемента-заголовка (это означает новый раздел), когда меняются метаданные элемента (это означает новую страницу или раздел) или когда сходство содержимого превышает порог.
4. Объединение меньших блоков: опционально объединять небольшие блоки, чтобы они были достаточно крупными для эффективного семантического поиска.

***Ключевые моменты***:
- Связные блоки: содержимое из одного и того же элемента документа остается вместе, благодаря чему блоки получаются более связными.
- Структурированные блоки: метод chunking может использовать структуру документа, в отличие от традиционных техник разбиения, которые делят текст по количеству токенов или символов.

## 8. Подходы к обработке изображений в данных

Для других документов, например PDF и изображений, информация является визуальной. Нам нужен Document Image Analysis (DIA), чтобы извлекать информацию о форматировании и текст из исходных изображений документа.
Сейчас у DIA есть два основных подхода:

- Document Layout Detection (DLD) использует модель обнаружения объектов, чтобы рисовать и размечать bounding boxes вокруг элементов макета на изображении документа.
- VisionTransformer (ViT) принимает изображение документа на вход и генерирует на выходе текстовое представление структурированного результата (например, JSON).

![alt text](<../../../07-第七章-Agent/assest/干货： 落地企业级RAG的实践指南/2.png>)

Если говорить конкретнее, шаги DLD таковы: 1) визуальное обнаружение: использовать модель компьютерного зрения (например, YOLOX или Detectron2) для распознавания и классификации bounding boxes. 2) извлечение текста: при необходимости использовать оптическое распознавание символов (OCR), чтобы извлечь текст из bounding boxes.
Примечание: для некоторых документов (например, PDF) текст можно извлекать напрямую из документа без OCR.

![alt text](<../../../07-第七章-Agent/assest/干货： 落地企业级RAG的实践指南/22.png>)

А ViT означает, что изображение документа передается в encoder, после чего decoder генерирует текстовый вывод. При этом Document Understanding Transformer (DONUT) — распространенная архитектура: ей не нужен OCR, она напрямую преобразует входное изображение в текст и даже может обучаться выводить сразу валидные JSON-строки!

## 9. Подходы к обработке таблиц в данных

Большинство RAG-сценариев сосредоточено на текстовом содержимом документов. В то же время некоторые отрасли (например, финансы и страхование) активно работают со структурированными данными, встроенными в неструктурированные документы. Чтобы поддерживать сценарии вроде вопросов и ответов по таблицам, нам нужно извлекать таблицы из документов.

В индустрии сейчас есть три технологии:
- Table Transformer: распознает bounding boxes ячеек таблицы и преобразует результат в HTML;
- Vision Transformer: использует модель визуального трансформера из предыдущего раздела (предобработка PDF и изображений), но с HTML на выходе;
- OCR + Table Parser: использует OCR для извлечения таблицы, а затем Table Parser для преобразования ее в структурированные данные.
