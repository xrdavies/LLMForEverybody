### 1.2 Три способности LLM applications

Три важнейшие способности LLM application — это архитектура, знания и модель.
- Архитектура
  - Декомпозиция задачи: модель за один раз выполняет одну небольшую задачу. Почему CoT работает? Причина в том, что большие модели не видели данных, где сложная задача выполняется целиком за один шаг, и не проходили pre-training на таких данных. Зато они видели данные, где сложные задачи разбиты на малые подзадачи, и именно это у них получается хорошо. Декомпозиция задачи — это фактически ручной CoT.
  - Retrieval augmentation: сделать так, чтобы найденные знания были достаточно полными и по возможности точными.
- Знания
  - Knowledge engineering: построение знаний разными способами, включая построение знаний для sparse retrieval вроде BM25, построение знаний для dense retrieval через semantic vector search, а также построение knowledge graph. Лучший результат можно получить только при hybrid retrieval по знаниям разных форматов.
- Модель
  - По возможности использовать более качественную базовую модель;
  - Продолжать обучение модели, внедрять знания вертикальной предметной области и выполнять fine-tuning под конкретный формат задачи;
  - Оптимизировать prompt.

### 2.2 Декомпозиция задач

В части декомпозиции задач обычно выделяют несколько Agent.

- На входе обычно находится Agent для распознавания намерения
  - Уточняет неоднозначность или недостающую информацию в вопросе пользователя;
  - Получает релевантную информацию через retrieval, чтобы помочь определить намерение;
  - Маршрутизирует конкретную задачу конкретному Agent.

- Последующие Agent больше сфокусированы на конкретных задачах
  - Выполняют уточнения на уровне задачи;
  - Получают через retrieval информацию, связанную с выполнением задачи;
  - Выполняют дальнейшую декомпозицию задачи. На этом этапе каждая задача может быть просто функцией/инструментом; если она становится очень сложной, это может быть и другой Agent.

### 3.3 Структурированные данные

- Для данных без семантической информации можно:
  - Построить MinHash LSH;
  - Построить knowledge graph.
- Для данных с семантической информацией можно дополнительно построить vector database.

### 3.4 Неструктурированные данные

- Построить inverted index для поддержки поиска методами вроде BM25;
- Построить vector database для semantic similarity vectors;
- Выполнить information extraction с помощью LLM, преобразовать данные в структурированный формат, а затем применить knowledge engineering для структурированных данных.

## 4. Оптимизация модели для LLM applications

### 4.1 Оптимизация prompt

- Prompt должен быть ясным и конкретным, без встроенной неоднозначности;
- Структурированный Prompt;
- Для задач, где reasoning дает более высокое качество, использовать CoT;
- Когда нужна максимальная производительность, использовать Self Consistency.

### 4.2 Fine-tuning модели

- Если нужно внедрить знания в модель, все равно придется делать CT (continued training); SFT (supervised fine-tuning) по-прежнему с трудом внедряет знания;
- Для некоторых конкретных небольших задач SFT на более маленьких моделях дает хороший эффект; для отдельных задач можно даже использовать более маленький BERT.

## 5. Процесс итеративной оптимизации LLM applications

### 5.1 Метрики оценки

![alt text](<../../../07-第七章-Agent/assest/LLM应用落地指南/14.png>)

- Для генеративных задач с feedback: главным образом это Text-to-SQL и Text-to-Code. Мы можем написать эталонный SQL или unit tests, чтобы проверить accuracy задачи.
- Для генеративных задач без feedback: остается только прибегать к оценке с помощью большой модели. Кто-то может посчитать, что так большая модель сама становится и участником, и судьей; но, во-первых, мы можем улучшить способность модели к оценке через fine-tuning, а во-вторых, при изменении Prompt меняется и смещение возможностей модели, поэтому этот метод все же применим. Конечно, у него неизбежно есть ограничения; в таком случае можно попробовать fine-tuning (см. 4.2).
