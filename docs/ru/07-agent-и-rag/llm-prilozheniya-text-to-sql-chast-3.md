## 4. Риски пользовательского опыта

![alt text](../../../07-第七章-Agent/assest/LLM应用落地指南/16.png)

У ошибок Text-to-SQL высокий эффект, потому что продукт рассчитан на пользователей, которые не занимаются разработкой данных, и они не понимают SQL. Если SQL окажется неверным, но вернет значение примерно того же порядка, пользователь может сделать ошибочный вывод, а влияние на бизнес будет довольно большим.

### 4.1 Снижение влияния ошибок

- Один очевидный вариант решения — попросить LLM объяснять сгенерированный SQL. Современные LLM довольно хорошо справляются с объяснением SQL. Так общее влияние ошибки можно снизить до среднего или даже низкого уровня. Высокий эффект ошибки возникнет только тогда, когда проблема появится уже у LLM, объясняющей SQL.

### 4.2 Снижение частоты ошибок

У Text-to-SQL частота ошибок будет высокой из-за всей сложности, о которой говорилось выше. Поэтому главный акцент — на снижении частоты ошибок. Помимо способов из предыдущего раздела, можно применять и инженерные приемы. Например:

- объединить данные в одну широкую таблицу, чтобы модель не выполняла слишком много join;
- заменить enum-значения на китайские, чтобы избежать галлюцинаций при преобразовании enum-значений моделью;
- заранее рассчитать некоторые очень сложные метрики;
- и так далее.

### 5.3 Разбор архитектуры

- Agent распознавания намерения
  - Поскольку это conversational application, нам обязательно нужен Agent распознавания намерения:
    - распознать намерение пользователя и выполнить routing;
    - одновременно ему нужно уточнять некоторые неоднозначные вопросы пользователя.
      - Пример: «помоги мне найти клиентов, которые купили продукт A и продукт B». У этой фразы есть две трактовки: первая — пересечение, то есть клиенты, купившие и продукт A, и продукт B; вторая — объединение, то есть клиенты, купившие продукт A или продукт B. Здесь Agent должен помочь нам уточнить смысл.
  - RAG для Agent распознавания намерения
    - Через retrieval augmentation извлекаются конкретные данные, которые помогают Agent определить, к какому намерению относится запрос. Если извлеченный контент показывает, что возможны оба намерения, нужно задать пользователю уточняющий вопрос.

- Text-2-SQL Agent
  - Уточнение
    - Уточнение относится к самой задаче.
      - Пример: пользователь хочет узнать продажи магазина KFC XXX, но магазинов с названием XXX может быть несколько: XXX 1, XXX 2. Тогда нужно попросить пользователя уточнить, какой именно магазин он имеет в виду.
      - Такое уточнение возможно только после того, как мы поняли, что это задача Text-to-SQL, а затем через recall значений из базы данных обнаружили неоднозначность.

  - Schema-Linking
    - Выбор таблиц и колонок, дополнение описаний, добавление кандидатных значений из базы данных для использования в where/having/case when и других выражениях.

  - Генерация SQL
    - self-consistency: самосогласованность снижает частоту ошибок;
    - Revise: самопроверка исправляет синтаксические ошибки.

  - Перевод SQL
    - Попросить Agent перевести SQL на естественный язык и вернуть этот перевод вместе с результатом выполнения SQL.
