Разрабатывать большую модель или использовать большую модель?

Недавно OpenAI, как ожидается, осенью выпустит новую AI-модель под кодовым названием «Strawberry». От задач, сфокусированных на математике, до обработки субъективных маркетинговых стратегий модель Strawberry демонстрирует удивительное разнообразие возможностей.

Обновления больших моделей мелькают перед глазами одно за другим, но во всей экосистеме больших моделей разделение труда на самом деле довольно четкое. Большинство специалистов в сфере больших моделей используют большие модели, а не разрабатывают foundation model.

## 1. Все более дорогой Pre-Training

Стоимость pre-training больших моделей многогранна и включает GPU, данные, хранилище и другие аспекты. Ниже подробно разобраны эти стороны:

**Стоимость GPU**:

Для обучения крупных моделей требуется огромное количество GPU-ресурсов. Например, обучение большой модели масштаба сотен миллиардов параметров может потребовать тысячи NVIDIA A100 GPU, а стоимость каждого GPU составляет около 10 000 долларов США. Если считать по такому масштабу, одна только стоимость GPU может достигать сотен миллионов долларов.

**Стоимость данных**:

Для обучения больших моделей нужны огромные объемы данных. Сбор, очистка, разметка и хранение данных — все это требует затрат. Например, pre-training dataset может проходить множество предварительных этапов, включая data crawling, очистку, преобразование и т. д. Эти этапы предполагают множество экспериментов, а объем обрабатываемых данных обычно более чем в 100 раз превышает объем формального training dataset.

**Стоимость хранилища**:

Баланс между производительностью и стоимостью системы хранения — важный фактор. Высокопроизводительные файловые системы вроде GPFS и Lustre обычно зависят от all-flash (NVMe) и высокопроизводительной сети, поэтому стоят дорого. Object storage дешевле, но может требовать дополнительных трудозатрат и времени на синхронизацию данных, миграцию, управление согласованностью и другие задачи.

**Стоимость дата-центра**:

Операционные расходы дата-центра включают электроэнергию, охлаждение, обслуживание и т. д. Эти расходы растут по мере увеличения числа GPU и масштаба дата-центра.

**Стоимость персонала**:

Для обучения крупных моделей нужна профессиональная команда инженеров и ученых, включая data engineers, AI researchers, software engineers и других специалистов. Зарплаты и бенефиты таких людей — еще один важный фактор затрат.

## 2. У вас действительно есть шанс заниматься pre-training большой модели?

Абсолютное большинство специалистов по большим моделям не будут заниматься разработкой foundation model.

Многие технологии pre-training вы могли видеть в технических блогах, о них вас могли спрашивать на собеседовании, но, возможно, вы никогда не будете применять их в реальной работе. Потому что pre-training слишком дорогой, и многим компаниям он просто не нужен.

Большинство специалистов по большим моделям используют большие модели, а не разрабатывают foundation model.

Если разделить приложения больших моделей по сложности, в основном получится пять направлений:

| Стратегия | Сложность| Требования к данным|
| :--- |:----:| :----: |
| Prompt Engineering|низкая|нет|
| Self-Reflection |низкая| нет|
| RAG|средняя|небольшой объем|
| Agent|средняя|небольшой объем|
| Fine-tuning |высокая|средний объем|

## 3. Prompt Engineering

Prompt Engineering — это искусство и наука оптимизации prompts для получения эффективного output. Оно включает проектирование, написание и изменение prompts, чтобы направлять AI-модель к генерации качественных, релевантных и полезных ответов.

## 4. Self-Reflection

В реальной работе я заметил, что многие коллеги не осознают важность Self-Reflection. На самом деле Self-Reflection — простая, но очень полезная стратегия.

Покажем на примере NL2SQL:

### Первое взаимодействие

```python
question = ''
prompt = f'{question}'
plain_query = llm.invoke(prompt)
try:
    df = pd.read_sql(plain_query)
    print(df)
except Exception as e:
    print(e)
```

Получив ошибку, мы можем через reflection над ошибкой улучшать наш вопрос, пока не получим нужный ответ.

### Reflection

```python
reflection = f"Question: {question}. Query: {plain_query}. Error:{e}, so it cannot answer the question. Write a corrected sqlite query."
```

### Второе взаимодействие

```python
reflection_prompt = f'{reflection}'
reflection_query = llm.invoke(reflection_prompt)
try:
    df = pd.read_sql(reflection_query )
    print(df)
except Exception as e:
    print(e)
```

## 5. RAG

Retrieval-Augmented Generation (сокращенно RAG) повышает точность и релевантность генерируемого текста, объединяя Large Language Model (LLM) и систему information retrieval. Такой подход позволяет модели перед генерацией ответа сначала извлечь релевантную информацию из авторитетной базы знаний, тем самым обеспечивая актуальность и профессиональность результата без переобучения самой модели.

Технология RAG важна потому, что она решает ряд ключевых проблем LLM, например выдачу ложной информации, генерацию устаревших сведений, зависимость от неавторитетных источников и неточные ответы из-за смешения терминов. Введение RAG позволяет извлекать информацию из авторитетных и заранее определенных источников знаний, усиливает контроль над генерируемым текстом и одновременно повышает доверие пользователей к AI-решениям.

## 6. Agent

Agent — это система, способная воспринимать окружение и на основе воспринятой информации принимать решения для достижения конкретной цели. Благодаря поддержке больших моделей Agent стал заметнее, чем когда-либо раньше.

### LangChain

Agent-фреймворки во главе с LangChain сейчас являются самыми широко используемыми open source-фреймворками в Китае. Изначальная идея дизайна LangChain заключалась в том, чтобы представлять workflow как DAG (directed acyclic graph, направленный ациклический граф); отсюда и произошло Chain.

По мере распространения идеи Multi-Agent и постепенного закрепления Agent-парадигмы Agent workflow становится все сложнее: в нем появляются циклы и другие условия. Для этого уже нужен Graph-подход, и так появился LangGraph.

## 7. Fine-tuning

По сравнению с ценой базовой большой модели, где счет легко идет на десятки тысяч карт, fine-tuning, возможно, является одним из немногих способов post-training большой модели, доступных обычным людям или компаниям.

Fine-tuning означает, что на основе pre-training-модели с помощью небольшого объема данных и вычислительных ресурсов модель дополнительно обучают, чтобы адаптировать ее к конкретной задаче или dataset.

![alt text](../../../07-第七章-Agent/assest/开发大模型or使用大模型/微调/0.png)

Fine-tuning делится на два типа: full fine-tuning и parameter efficient fine-tuning.

- Full fine-tuning: при full fine-tuning обновляются все параметры модели; обычно такой метод требует большого объема данных и вычислительных ресурсов, а также более длительного времени обучения.

### PEFT

Parameter-Efficient Fine-Tuning (сокращенно PEFT) — это техника fine-tuning для крупных pre-training-моделей, например large language models. Ее цель — уменьшить количество обучаемых параметров, тем самым снизив вычислительные затраты и стоимость хранения при сохранении или повышении качества модели.

PEFT адаптирует модель к конкретной downstream task, дообучая лишь небольшую часть параметров модели, а не всю модель целиком. Такой подход особенно полезен в условиях ограниченных hardware-ресурсов, а также когда крупной модели нужно быстро адаптироваться к множеству задач.

![alt text](../../../07-第七章-Agent/assest/开发大模型or使用大模型/微调/12.png)

У PEFT есть несколько распространенных методов:

- выбор подмножества параметров: выбрать для fine-tuning небольшую часть параметров модели, обычно параметры последних нескольких слоев;
- репараметризация: использовать низкоранговое представление для репараметризации весов модели; типичный представитель — метод LoRA;
- добавление параметров: добавить в модель обучаемые слои или параметры; типичный представитель — метод Prompt-tuning.
