FlashAttention V2 уменьшает объем вычислений и обращений к памяти, сохраняя точность и эффективность алгоритма, и обеспечивает более быстрое вычисление Attention. Благодаря этим оптимизациям скорость forward propagation в версии V2 на A100 GPU выросла примерно в 2 раза и достигла 50%-73% от теоретического вычислительного пика.

## 1. FlashAttention v1

> $Attention(Q, K, V) = softmax(\frac{QK^T}{\sqrt{d_k}})V$

FlashAttention не реализует матрицы X и A в глобальной памяти, а объединяет все вычисление из приведенной выше формулы в одном CUDA kernel. Для этого нужно разработать алгоритм, который аккуратно управляет on-chip memory (например, потоковый алгоритм), потому что shared memory (SRAM) у NVIDIA GPU невелика.

В классических алгоритмах вроде умножения матриц используется tiling, чтобы объем on-chip memory не превышал аппаратных ограничений. Этот подход работает потому, что сложение ассоциативно и позволяет разложить умножение всей матрицы на сумму многих tiled matrix multiplication.

## 2. Что обновилось в FlashAttention v2

![alt text](<../../../01-第一章-预训练/assest/FlashAttention v2相比于v1有哪些更新？/1.png>)

- Уменьшено количество non-matmul FLOPs (устранены прежние частые операции rescale). Хотя non-matmul FLOPs составляют лишь небольшую долю от общего числа FLOPs, они выполняются дольше, потому что у GPU есть специализированные блоки для матричного умножения, и их пропускная способность до 16 раз выше, чем у нематричных операций. Поэтому важно уменьшать non-matmul FLOPs и выполнять как можно больше matmul FLOPs.

- Предложена параллелизация по измерению длины последовательности. Этот метод повышает утилизацию GPU, когда входная последовательность очень длинная (в таких случаях batch size обычно небольшой). Даже для одного head вычисления выполняются параллельно между разными thread block.

- Внутри одного блока вычисления attention работа распределяется между разными warp одного thread block, чтобы уменьшить коммуникации и чтение/запись shared memory.

## 3. Подробно о первом обновлении

Уменьшено количество non-matmul FLOPs (устранены прежние частые операции rescale). Хотя non-matmul FLOPs составляют лишь небольшую долю от общего числа FLOPs, они выполняются дольше, потому что у GPU есть специализированные блоки для матричного умножения, и их пропускная способность до 16 раз выше, чем у нематричных операций. Поэтому важно уменьшать non-matmul FLOPs и выполнять как можно больше matmul FLOPs.

### non-matmul vs. GEMM

***non matrix multiply***

Нематричное умножение: операции за пределами матричного умножения, например сложение, умножение, деление и т. д. Например, операция `rescale` использует универсальное аппаратное обеспечение GPU: CUDA Core.

***General Matrix Multiply***

Общее матричное умножение: один из способов реализации матричного умножения, который использует специализированное аппаратное обеспечение GPU: Tensor Core.

### CUDA Core vs. Tensor Core

Нематричные операции выполняются дольше, потому что у GPU есть специализированный вычислительный блок для матричного умножения - Tensor Core, чья пропускная способность до 16 раз выше, чем у нематричных операций.
На рисунке ниже приведено описание A100. Видно, что вычислительная мощность CUDA Core в single precision TF16 составляет 19.5 TFLOPS, а вычислительная мощность Tensor Core - 312 TFLOPS. Разница достигает 16 раз.

### Операция Rescale

![alt text](<../../../01-第一章-预训练/assest/FlashAttention v2相比于v1有哪些更新？/2.png>)

Версия V2 изменила алгоритм: она уменьшила количество floating-point operations для нематричных операций, сохранив выход неизменным. В исходном FlashAttention (то есть V1) на каждой итерации каждого block нужно выполнять операцию rescale, а она включает деление.

В V2 эта операция rescale откладывается до конца цикла и выполняется только один раз; каждое вычисление может сэкономить одну операцию деления. Такая корректировка возможна потому, что на каждой итерации достаточно гарантировать, что числитель масштабирован к правильному значению, а знаменатель посчитан правильно. Эта оптимизация уменьшает объем вычислений и повышает эффективность.


## 4. Подробно о втором обновлении
Предложена параллелизация по измерению длины последовательности. Этот метод повышает утилизацию GPU, когда входная последовательность очень длинная (в таких случаях batch size обычно небольшой). Даже для одного head вычисления выполняются параллельно между разными thread block.

### Аппаратная часть GPU
  - Streaming Processor (SP): базовый вычислительный блок; начиная с архитектуры Fermi его называют CUDA core.
  - Streaming MultiProcessor (SM): один SM состоит из нескольких CUDA core (SP). В разных архитектурах GPU у каждого SM разное количество CUDA core; например, в архитектуре Pascal один SM содержит 128 CUDA core.

### Программная часть GPU
  - thread: параллельная CUDA-программа выполняется множеством thread; thread - базовая единица выполнения (the basic unit of execution);
  - warp: один warp обычно содержит 32 thread. Thread внутри одного warp могут одновременно выполнять одну и ту же инструкцию, реализуя SIMT (single instruction, multiple threads) параллелизм. Warp - минимальная единица планирования на SM (the smallest scheduling unit on an SM), один SM может одновременно обрабатывать несколько warp;
  - thread block: один thread block может содержать несколько warp. Thread внутри одного block могут синхронизироваться и общаться через shared memory. Thread block - минимальная единица выполнения на GPU (the smallest unit of execution on the GPU). Thread одного warp обязательно находятся в одном block. Если количество thread в block не кратно размеру warp, то в лишнем warp останутся inactive thread. То есть даже если thread в warp не хватает, аппаратно warp все равно будет дополнен до полного размера, просто часть thread будет inactive, но ресурсы SM они все равно потребляют.

Второе важное обновление V2 по сравнению с V1 - добавление параллельного вычисления по измерению длины последовательности. Цель - повысить утилизацию SM (Streaming Multiprocessor) GPU, особенно при обработке длинных последовательностей. В V1 вычисление сначала параллелилось по batch и числу head, а затем выполнялось последовательно по длине последовательности. Это значит, что при большой длине последовательности все доступные SM могли использоваться не полностью, потому что каждый block обрабатывал только один фрагмент последовательности.

В V2 параллелизация по измерению длины последовательности позволяет эффективнее распределять вычислительные задачи между большим числом block и тем самым полнее использовать ресурсы GPU. Конкретно V2 вводит понятие num_m_block и дополнительно разбивает матрицу Q на небольшие блоки вдоль длины последовательности, причем каждый такой блок обрабатывается отдельным block. Так каждый block может независимо вычислять свою часть выходного результата, что уменьшает зависимости и коммуникационные накладные расходы между разными block.

> Это немного похоже на идею continous batching.

## 5. Подробно о третьем обновлении

![alt text](<../../../01-第一章-预训练/assest/FlashAttention v2相比于v1有哪些更新？/4.png>)

В V2 порядок циклов изменен: Q вынесена во внешний цикл, а K и V - во внутренний. Каждый thread block отвечает за вычисление части выходной матрицы O. Такой дизайн позволяет каждому thread block выполнять вычисления независимо и снижает зависимости и потребность в коммуникации между thread block. Кроме того, в версии V2 в forward propagation дополнительно уменьшено количество floating-point operations для нематричных операций, чтобы максимально использовать специализированные вычислительные блоки на GPU, такие как Tensor Cores на Nvidia GPU, и тем самым максимизировать throughput GPU.
Кроме того, в backward propagation версия V2 тоже оптимизирована: она использует похожую стратегию разбиения на блоки для оптимизации вычислений и доступа к памяти, повышая эффективность и производительность. Благодаря этому FlashAttention V2 может обеспечить более высокий параллелизм, сократить лишние вычисления и обращения к памяти и тем самым улучшить общую вычислительную производительность.
