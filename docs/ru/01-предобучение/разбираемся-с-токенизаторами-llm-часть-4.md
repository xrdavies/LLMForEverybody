Каждый день за 5 минут: разбираемся с Tokenizer в LLM, часть 4: Unigram

В SentencePiece часто используется алгоритм Unigram. Это алгоритм токенизации, который применяют такие модели, как AlBERT, T5, mBART, Big Bird и XLNet.

![alt text](../../../01-第一章-预训练/assest/搞懂大模型的分词器（四）/1.png)

По сравнению с BPE и WordPiece, Unigram устроен иначе: он начинает с большого словаря, а затем удаляет из него token, пока не достигнет нужного размера словаря.

На каждом шаге обучения алгоритм Unigram вычисляет loss корпуса при текущем словаре.

Затем для каждого token в словаре алгоритм считает, насколько увеличится общий loss, если удалить этот token, и ищет token с минимальным ростом loss.

Такие token меньше влияют на общий loss корпуса, поэтому в некотором смысле они «менее нужны» и являются лучшими кандидатами на удаление.


## Построение базового словаря

Первый шаг алгоритма Unigram — построить базовый словарь, содержащий все возможные token.

Здесь мы больше не будем использовать пример с olympic, а возьмем официальный пример Hugging Face. Причина проста: автор ленится считать столько вероятностей вручную.

Предположим, у нас есть корпус, который содержит такие слова:

```plaintext
("hug", 10), ("pug", 5), ("pun", 12), ("bun", 4), ("hugs", 5)
```

Числа показывают, сколько раз слово встречается.

Тогда наш базовый словарь выглядит так; он содержит все sub-word:

```plaintext
["h", "u", "g", "hu", "ug", "p", "pu", "n", "un", "b", "bu", "s", "hug", "gs", "ugs"]
```

## Unigram-модель

Сначала посчитаем частоты всех subword в этом базовом словаре:

```plaintext
("h", 15) ("u", 36) ("g", 20) ("hu", 15) ("ug", 20) ("p", 17) ("pu", 17) ("n", 16) ("un", 16) ("b", 4) ("bu", 4) ("s", 5) ("hug", 15) ("gs", 5) ("ugs", 5)
```
Сумма частот равна 210, значит вероятность одного subword, например 'ug', равна 20/210.

Unigram-модель — самая простая языковая модель. Она предполагает, что каждое слово независимо, поэтому мы можем напрямую считать вероятность каждого слова.

Например, вероятность одного способа токенизации слова 'pug', ['p','u','g'], равна:

$$p(['p','u','g']) = P('p')*P('u')*P('g') = 5/210 * 36/210 * 20/210 = 0.000389$$

А вероятность другого способа токенизации 'pug', ['pu','g'], равна:

$$p(['pu','g']) = P('pu')*P('g') = 5/210 * 20/210 = 0.0022676$$

То есть для слова 'pug' мы можем посчитать вероятности всех вариантов токенизации и выбрать вариант с максимальной вероятностью.

```plaintext
['p','u','g'] 0.000389
['pu','g'] 0.0022676
["p", "ug"] : 0.0022676
```
Мы выберем ['pu','g'] как токенизацию слова 'pug'. Если вероятности совпадают, можно выбрать первый вариант.

Так мы можем tokenizer-ом обработать все word и получить тот самый большой словарь, о котором говорили выше.

На практике есть небольшая проблема: полный перебор слишком дорогой. Настолько дорогой, что автор этой статьи сходу тоже не хочет все это считать.

>Вопрос: если в нашем word есть n character, сколько всего вариантов токенизации получится при полном переборе? Добро пожаловать в комментарии.

На практике для решения этой задачи перебора можно использовать алгоритм Витерби (Viterbi).

Алгоритм Витерби не входит в область обсуждения этой статьи; если интересно, можно самостоятельно почитать материалы.

## Удаление token
Теперь у нас уже есть большой словарь. Дальше нужно удалить часть token, пока мы не достигнем целевого размера словаря.

Логика удаления такая же, как при сокращениях в компании: кто меньше всего влияет на общую систему, тот и удаляется.

Важно: базовые character удалять нельзя. Они нужны нам, чтобы генерировать OOV.

Для корпуса выше:
```plaintext
("hug", 10), ("pug", 5), ("pun", 12), ("bun", 4), ("hugs", 5)
```

Предположим, после предыдущего шага мы уже токенизировали каждое word и получили такие сегментации и score:

```plaintext
"hug": ["hug"] (score 0.071428)
"pug": ["pu", "g"] (score 0.007710)
"pun": ["pu", "n"] (score 0.006168)
"bun": ["bu", "n"] (score 0.001451)
"hugs": ["hug", "s"] (score 0.001701)
```

>Важно: в приведенном tokenizer опущены отдельные character. Например, токенизация "hug" на самом деле включает ["h","u","g","hug"], но для простоты мы опустили ["h","u","g"].

Теперь нужно посчитать влияние каждого token на общий результат. Этот loss — negative log likelihood, отрицательное логарифмическое правдоподобие, рассчитанное по этим score.

Начальный loss:

$$10 * (-log(0.071428)) + 5 * (-log(0.007710)) + 12 * (-log(0.006168)) + 4 * (-log(0.001451)) + 5 * (-log(0.001701)) = 169.8$$

Предположим, token, который мы хотим удалить, — это 'hug'. Тогда затронуты токенизации hug и hugs. После обновления таблицы token без 'hug' score становятся такими:
    
```plaintext
"hug": ["hu", "g"] (score 0.006802)
"hugs": ["hu", "gs"] (score 0.001701)
```

Изменение loss:

$$hug：- 10 * (-log(0.071428)) + 10 * (-log(0.006802)) = 23.5 $$

$$hugs：- 5 * (-log(0.001701)) + 5 * (-log(0.001701)) = 0 $$

Общее изменение равно 23.5.

Мы перебираем все token, находим token с минимальным увеличением loss и удаляем его.

Повторяем эти шаги, пока не достигнем целевого размера словаря.
